# Product MDM 接入规范

## 1. 概述

### 文档用途

本文档定义 edd-mdm Product MDM 子域的跨系统接入契约，是所有消费 edd-mdm 主数据的下游系统的单一事实来源（SSOT）。

### 覆盖范围

- **实体类型**：Brand（品牌）、CarLine（车系）、Platform（平台）
- **接入方式**：Kafka 事件订阅、Feign 接口调用
- **适用系统**：VMD、订单服务、销售配置器、BI 系统等所有需要消费产品主数据的下游

### 文档版本

- **当前版本**：1.0.0
- **版本策略**：语义化版本（主版本.次版本.修订号）
  - 主版本（X.0.0）：破坏性变更，下游必须改造
  - 次版本（x.Y.0）：新增字段或接口，向后兼容
  - 修订号（x.y.Z）：文档修正，无契约变更

## 2. Kafka 事件契约

### 2.1 Topic 命名规约

```
mdm.product.<entity>.<eventType>
```

**entity 取值**：brand / carline / platform  
**eventType 取值**：created / updated / deactivated

**完整 Topic 列表**：

| Topic | 说明 |
|-------|------|
| mdm.product.brand.created | 品牌创建 |
| mdm.product.brand.updated | 品牌更新 |
| mdm.product.brand.deactivated | 品牌失效 |
| mdm.product.carline.created | 车系创建 |
| mdm.product.carline.updated | 车系更新 |
| mdm.product.carline.deactivated | 车系失效 |
| mdm.product.platform.created | 平台创建 |
| mdm.product.platform.updated | 平台更新 |
| mdm.product.platform.deactivated | 平台失效 |

### 2.3 Partition 策略

- **策略**：按 entity code hash 分配 Partition
- **目的**：保证同一实体的事件有序（created → updated → deactivated）
- **实现**：Kafka Producer 发送时指定 Key = entity code

```
ProducerRecord(topic, key=entityCode, value=eventPayload)
```

### 2.4 顺序保证

- **同 Partition 内有序**：同一实体的所有事件保证有序
- **跨 Partition 无序**：不同实体的事件可能乱序，下游需自行处理

### 2.5 重试策略

| 参数 | 默认值 | 说明 |
|------|--------|------|
| 重试次数 | 3 | 超出后投递死信队列 |
| 重试间隔 | 指数退避（1s, 2s, 4s） | 避免重试风暴 |
| 告警阈值 | 重试次数 >= 2 | 触发告警通知 |

### 2.6 死信队列

- **Topic**：`mdm.product.dlq`
- **Payload**：包含原始事件 + 错误信息 + 重试记录
- **处理**：人工介入排查，修复后可重新投递

### 2.7 幂等键

- **幂等键**：eventId（UUID）
- **下游处理**：建议下游记录已处理的 eventId，重复事件直接忽略
- **实现建议**：使用 Redis 或数据库记录 eventId，设置 TTL（建议 7 天）

### 3.2 FallbackFactory 行为约定

| 场景 | Fallback 行为 |
|------|---------------|
| 全量快照调用失败 | 返回空列表 `[]`，记录 WARN 日志 |
| 单点查询调用失败 | 返回 `null`，记录 WARN 日志 |
| 超时 | 同上，记录超时异常信息 |
| 熔断 | 同上，记录熔断异常信息 |

**日志格式**：

```
WARN [Feign Fallback] service=edd-mdm, method=listAll, entity=brand, error={error message}
```

## 4. 数据一致性约定

### 4.1 下游 Upsert 逻辑

下游系统消费事件后，应按以下逻辑更新本地副本：

```sql
-- 伪代码
IF event.version > local.external_version THEN
    INSERT OR UPDATE local_table
    SET ... = event.payload,
        external_version = event.version,
        last_sync_time = now()
    WHERE external_ref_id = event.entityId
ELSE
    -- 忽略旧版本事件
    IGNORE
END IF
```

**关键字段映射**：

| 下游本地字段 | MDM 事件字段 | 说明 |
|-------------|-------------|------|
| external_ref_id | entityId | MDM 侧业务主键（code） |
| external_version | version | MDM 侧版本号 |
| source | 固定值 'MDM' | 数据来源标识 |
| last_sync_time | 事件消费时间 | 最后同步时间 |

### 4.2 Bootstrap 时机与策略

#### 时机

| 场景 | 触发方式 |
|------|----------|
| 系统首次上线 | 手动触发全量拉取 |
| 数据修复 | 手动触发全量拉取 |
| 服务重启 | 可选：启动时检测本地数据为空则自动拉取 |

#### 策略

1. **全量拉取**：调用 Feign 接口分页拉取全量数据
2. **本地覆盖**：使用 upsert 逻辑覆盖本地数据
3. **版本校验**：仅当 event.version > local.external_version 时更新

**Bootstrap 流程**：

```
1. 调用 GET /api/service/brand/v1/listAll?page=1&size=500
2. 循环拉取直至 page > total/size
3. 对每条记录执行 upsert 逻辑
4. 重复 1-3 处理 Series、Platform
```

### 4.3 对账机制

**本期不实现**，留待后续 CR。

**建议方案**：

- 每日定时任务对比 MDM 与本地数据
- 输出 diff 报表（差异记录列表）
- 支持手动触发对账

## 5. Brand-CarLine 关联约定

### 设计决策

CarLine 事件 payload 必须包含 `brandCode` 字段，下游系统**不得**通过 join Brand 表获取品牌信息。

### 原因

1. **解耦**：下游无需维护 Brand 表即可处理 CarLine 数据
2. **性能**：避免跨表 join，提升查询效率
3. **简化**：事件自包含，减少依赖

### 下游处理

```java
// 消费 CarLine 事件时
String brandCode = event.getPayload().getBrandCode();
// 直接使用 brandCode，无需查询 Brand 表
```

## 6. 错误码总表

### 6.1 edd-mdm 自身错误码（段位 807XXX）

| 错误码 | 说明 | HTTP 状态码 | 场景 |
|--------|------|------------|------|
| 807001 | 业务主键（code）已存在 | 409 | 创建时 code 重复 |
| 807002 | 记录不存在 | 404 | 更新/删除/查询时记录不存在 |
| 807003 | 状态不允许删除 | 400 | 删除非 DRAFT 状态记录 |
| 807004 | 生效期无效 | 400 | effectiveFrom > effectiveTo |
| 807005 | 引用的 Brand 不存在或状态无效 | 400 | 创建 CarLine 时 brandCode 无效 |
| 807006 | 状态不允许失效 | 400 | 失效非 ACTIVE 状态记录 |
| 807099 | 系统内部错误 | 500 | 未知异常 |

### 6.2 跨系统约定错误码

| 错误码 | 说明 | 处理建议 |
|--------|------|----------|
| EVENT_VERSION_CONFLICT | 事件版本冲突 | 忽略旧版本事件 |
| EVENT_PAYLOAD_INVALID | 事件 Payload 格式错误 | 记录日志，人工介入 |
| FEIGN_TIMEOUT | Feign 调用超时 | 重试或使用 Fallback |
| FEIGN_CIRCUIT_OPEN | Feign 熔断 | 使用 Fallback，等待恢复 |

## 7. 版本演进规约

### 7.1 版本号规则

- **主版本（X.0.0）**：破坏性变更
  - 删除字段
  - 修改字段类型
  - 修改 Topic 命名
  - 修改接口 Path
- **次版本（x.Y.0）**：向后兼容变更
  - 新增字段
  - 新增接口
  - 新增 Topic
- **修订号（x.y.Z）**：文档修正
  - 错别字修正
  - 说明补充
  - 格式调整

### 7.2 变更流程

1. **提出变更**：在 edd-mdm 项目提交 CR（Change Request）
2. **影响评估**：评估对下游系统的影响
3. **文档更新**：更新本文档，记录 Changelog
4. **通知下游**：通过邮件/钉钉通知下游系统负责人
5. **灰度发布**：先在测试环境验证，再上线生产
6. **版本兼容期**：主版本变更需保留旧版本接口至少 30 天

### 7.3 新增事件规约

新增事件类型时：

1. 遵循 Topic 命名规约：`mdm.product.<entity>.<eventType>`
2. Payload 结构保持通用字段（eventId, eventType, occurredAt, entityId, version）
3. 新增字段放在 payload 内，不影响已有字段
4. 下游消费时忽略未知字段（向前兼容）

### 7.4 新增字段规约

新增字段时：

1. 字段必须可选（nullable），不得破坏现有消费方
2. 下游消费时忽略未知字段
3. 文档次版本号 +1

### 7.5 废弃规约

废弃字段或接口时：

1. 先标记 `@Deprecated`，文档说明替代方案
2. 保留至少 30 天兼容期
3. 兼容期后删除，主版本号 +1
